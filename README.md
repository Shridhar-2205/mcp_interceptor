# mcp_lab — MCP client, server, and interceptors in the middle

A tiny lab that shows an **MCP client** and **MCP server** talking over stdio,
with an **interceptor** (man-in-the-middle proxy) that can sit between them.
There are two interceptors — one that only **logs** traffic, and one that
**modifies another `.py` file** from the traffic — plus a **direct** mode with
no interceptor at all.

```
mcp_client.py  <--stdio-->  [interceptor]  <--stdio-->  mcp_server.py
                                  │
                                  └── observes / logs / code-gens
```

## How the interception works

MCP's stdio transport is **newline-delimited JSON-RPC** — each message is one
line. The client launches the interceptor *as if it were the server*; the
interceptor launches the real server and pumps bytes both ways, logging (or
acting on) each line and forwarding it verbatim. Neither side needs to know it's
there. Diagnostics go to **stderr** / a file only — never stdout, because stdout
is the live JSON-RPC channel back to the client.

## Files

| File | What |
|---|---|
| `mcp_server.py` | MCP server (stdio) with `add`, `echo`, `slugify` tools |
| `mcp_client.py` | MCP client; picks the mode via a flag (see Scenarios) |
| `interceptor.py` | transparent stdio proxy that **logs** client↔server traffic |
| `interceptor_modify.py` | active proxy that **rewrites another `.py` file** (`replay_calls.py`) from intercepted tool calls |
| `replay_calls.py` | auto-generated by `interceptor_modify.py` (git-ignored); a runnable script that replays the captured calls |
| `intercept.log` | auto-generated by `interceptor.py` (git-ignored); full JSON-RPC transcript |
| `requirements.txt` | `mcp>=1.28.1` (official Python SDK) |

## Setup

```bash
cd mcp_lab
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
```

## Scenarios

| Scenario | Command | Path | Side effect |
|---|---|---|---|
| **1. Logging interceptor** (default) | `python mcp_client.py` | client → `interceptor.py` → server | writes `intercept.log` |
| **2. Direct (no interceptor)** | `python mcp_client.py --direct` | client → server | none |
| **3. File-modifying interceptor** | `python mcp_client.py --modify` | client → `interceptor_modify.py` → server | rewrites `replay_calls.py` |

In every scenario the client makes the same 3 tool calls and prints:

```
[client] tools: ['add', 'echo', 'slugify']
[client] call add({'a': 2, 'b': 3}) -> 5
[client] call echo({'text': 'hello mcp'}) -> hello mcp
[client] call slugify({'text': 'On Call L9'}) -> on-call-l9
```

### Scenario 1 — Logging interceptor (default)

Every JSON-RPC message is logged (summary to stderr, full raw to `intercept.log`)
and forwarded unchanged.

```bash
python mcp_client.py
cat intercept.log      # full transcript: initialize, tools/list, each tools/call + result
```

stderr shows the live flow:

```
[interceptor] proxying stdio to: .../mcp_server.py
[interceptor] full transcript -> .../intercept.log
client -> server  method=initialize id=0
server -> client  id=0 result=ok
client -> server  method=tools/call id=2 tool=add
server -> client  id=2 result=ok
...
```

### Scenario 2 — Direct (no interceptor)

The client talks to the server directly — nothing sits in the middle, nothing is
logged. Useful as a baseline to confirm the interceptors are transparent.

```bash
python mcp_client.py --direct
```

### Scenario 3 — File-modifying interceptor

Same transparent proxy, but each intercepted `tools/call` is recorded and the
interceptor **(re)writes `replay_calls.py`** into a runnable script that replays
those exact calls. So intercepting the traffic edits another Python file, live:

```bash
python mcp_client.py --modify
```

```
[modify] proxying stdio to: .../mcp_server.py
[modify] will rewrite: .../replay_calls.py
[modify] recorded tool call #1 (add) -> rewrote replay_calls.py
[modify] recorded tool call #2 (echo) -> rewrote replay_calls.py
[modify] recorded tool call #3 (slugify) -> rewrote replay_calls.py
```

Then run the generated file to replay the captured calls against the server:

```bash
python replay_calls.py
# replay add({'a': 2, 'b': 3}) -> 5
# replay echo({'text': 'hello mcp'}) -> hello mcp
# replay slugify({'text': 'On Call L9'}) -> on-call-l9
```

It still forwards every message verbatim (it code-gens as a *side effect*; it
does not block or alter the in-flight call). The output path is a fixed filename
and arguments are written as Python **literals** (via `repr` of JSON-parsed
values), so intercepted data can't redirect the write or inject code.

### Bonus — proxy any other MCP stdio server

Both interceptors accept a target command, so you can tap a different server:

```bash
python interceptor.py <server-command> <args...>
python interceptor_modify.py <server-command> <args...>
```

## Security notes (MCP guidelines applied)

- **stdio transport** for every part — pipe-based, no network listener, so no
  DNS-rebinding / CORS / CSRF surface (recommended for local MCP).
- **Single-purpose tools** with explicit **input validation** and length caps;
  no "do anything" tool; least privilege (no fs/network side effects).
- Interceptors are **local observability / codegen** aids: they log or write to
  a local file with a fixed path and treat wire data as literals (never as code
  or as a path). In a real deployment, redact sensitive fields before logging
  and restrict access to `intercept.log` / generated files, per the logging
  guidelines.
